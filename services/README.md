# Services Layer

The mini-program pages should call service modules instead of reading mock data directly.

Default local route:

```text
pages -> services/* -> utils/mock-data.js / utils/mock-user.js
```

Cloud route:

```text
pages -> services/* -> services/cloud-data.js -> WeChat cloud database
```

## Rules

- Pages must not `require("../utils/mock-data")` or `require("../utils/mock-user")` directly.
- Do not re-enable native `tabBar`; keep using `components/bottom-nav`.
- Do not restore the old home/house button flow.
- Permission role checks must use `role/roleLabel`; do not use `isManager` to infer department-manager privileges.
- Mock role preview is allowed only outside `release`.
- Write operations should record audit entries through `audit-service.js`.
- QB in this mini-program is read-only. Create/transfer/progress/close actions belong to DingTalk; mini-program services only keep readonly compatibility returns plus `upsertQbFromDingTalk()` / `syncQbFromDingTalk()`.
- Excel import uses one standard template, exact header-name recognition and configurable field mapping.
- Excel import must not guess arbitrary headers.
- Device model and area are device basic fields, not electrical parameters.
- All getProject/getDevice/getUser entry points must enforce strict equality checks.
- Project detail process expansion must read through `getDevicesByProject()` / `getProcessesByDevice()`; pages must not build process rows from mock data.
- Task views and parameter-library searches must use service fallbacks (`filterTasksByView()`, `getDeviceParams()`) so local cloud snapshots missing `processMap` or `electricalParamValues` do not make the UI blank.
- Project detail processes, task views, and parameter-library/export data are a linked regression group. Any change touching devices, process tasks, parameters, cloud snapshots, import, permissions, or page data loading must verify all three service paths together: `getDevicesByProject()` / `getProcessesByDevice()`, `filterTasksByView("mine")`, and `listParamDevices()` / `getDeviceParams()` / `searchParams()`.
- Department-manager device dispatch uses existing pending-assignment task `rowKey` values and only lists same-department ordinary employees as assignees.
- Progress submission is split by role: ordinary employees submit per-device process progress and the service records it as `已完成`; department managers submit project-level department progress through `submitProjectDepartmentProgress()` and must not submit per-device progress.
- Department-manager project-level progress submission must be scoped to the manager's own business department only. Do not show submit buttons for other departments' processes.
- Department-manager project-level submit entry must be generated by `data-service.getDepartmentProjectSubmitOptions(projectNo, user)`. Show the entry only when that project has at least one enabled device process in the current manager's business department; do not infer it from `isManager` or from page-side mock traversal.
- Department-manager project-level progress must call `getProjectDepartmentProgressPreview()` first. If unfinished devices exist, the page must show the `未完成设备 / 负责人 / 预计完成时间` blocking table with only `已知晓`, then return to the previous page; `submitProjectDepartmentProgress()` must fail and write nothing until every related device process is completed.
- Process `actualStart` is a stored field from seed data, Excel import, or the progress-submit form; it is not automatically inferred.
- Progress-submit `actualStartDate` should default from dispatch dates: project-level department submission reads the project `adminOrderDate` (shown as 下单时间 / progress-manager dispatch time) through `getProjectDepartmentDispatchDate()` before falling back to explicit `dispatchedAt` or process `actualStart`; ordinary employee device-level submission reads employee assignment `assignedAt` through `getDeviceProcessAssignmentDate()`, then falls back to existing process `actualStart` only when assignment dates are missing.
- Current mock users/processes follow `outputs/relationship-review-20260630/人员角色部门工序核对表.xlsx`: 陈尚杰 is the 电气设计部 department manager; 彭博 is a 结构设计部 ordinary employee; process names include `ERP录入` and `物料采购`.
- The project-list header should only show the main title; do not add a static subtitle such as `除湿机项目试点`. Project records such as `C26-0501` must not be hidden or removed just because the title area should be simplified.
- Legacy names are normalized in services: `采购物料` -> `物料采购`, `电工房` -> `电气电控车间`, `结构班组/电气班组` -> `生产装配`, `生产部` -> `工艺部门`.

## Cloud Migration

### Transport modes

`services/cloud-config.js` uses `transport: "auto"`:
- Cloud function `appStore` deployed → calls go through `wx.cloud.callFunction`
- Not deployed → falls back to direct cloud database access

### Data persistence strategy

```
persistStore:
  1. writeStoreSnapshot (appStores/production-progress-store)
  2. writeDetailCollections (users/departments/projects/devices/qbRecords/processTasks/importLogs/operationLogs)
  Both succeed → syncState = "synced"

loadStoreFromCloud:
  1. readDetailCollections (parallel read from all detail collections → reassemble snapshot)
  2. If detail collections empty → readStoreSnapshot (snapshot fallback)
  3. If all empty and seedFromMockWhenEmpty → persistStore("seed")
```

`readDetailCollections()` uses `_sourceKey` equality conditions when reading detail collections, so DevTools does not warn about empty-condition full-table scans. Legacy or empty detail collections still fall back to the main store snapshot.

### Version optimistic lock

`storeVersion` tracks the local store version. Incremented on each persistStore. Written alongside the snapshot. On load, restored from local cache then overwritten by cloud data.

### Cloud function: appStore

5 actions: `getStore`, `saveStore`, `health`, `login`, `downloadTemplateFile`
- `login`: matches openid against `users` collection → store snapshot fallback → auto-link by name/phone
- `downloadTemplateFile`: validates and downloads the standard progress-import template from cloud storage.

### P0 cloud launch checklist

1. Enable cloud dev in WeChat DevTools → get envId → set in `cloud-config.js`
2. Create 10 collections in cloud console
3. Deploy `cloudfunctions/appStore`
4. "我的" page → Diagnose → all green + transport: function
5. Sync → close/reopen → Pull verify
6. Configure DB security rules
